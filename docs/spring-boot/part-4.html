STEP 4 — INTERNALS

It's not magic.
It's just clever.

Spring Boot's auto-configuration feels like magic the first time. It isn't. In this step you'll see exactly how it detects what you need and wires it up — so you can debug it when it goes wrong.

The foundation

The Application Context — Spring's bean factory

Everything in Spring revolves around one idea: Spring manages your objects. Instead of you calling new ProductService(), Spring creates it, configures it, and hands it to you when you need it. These managed objects are called beans.

The Application Context is the container that holds all beans. Click each bean below to see what it does and who created it.

Dependency injection

Spring wires your objects together

Once Spring has all beans in the container, it injects them into classes that declare they need them. This is Dependency Injection (DI) — the pattern that makes your code testable and decoupled. Toggle to see the difference it makes.

// You manually create and wire everything
// Every class must know how to build its own dependencies. // Change DataSource? Update every class that uses it. public class ProductController { private final ProductService productService; public ProductController() { // You build the whole graph yourself DataSource ds = new HikariDataSource(config); EntityManager em = Persistence.createEntityManager(ds); ProductRepository repo = new ProductRepository(em); this.productService = new ProductService(repo); } }

Testing this is painful — you can't swap in a fake repository without rewriting the constructor.

// Spring injects dependencies automatically
// Declare what you need. Spring provides it. // Change DataSource config? Only application.properties changes. @RestController public class ProductController { private final ProductService productService; // Constructor injection — Spring calls this automatically public ProductController(ProductService productService) { this.productService = productService; } }
// Testing becomes trivial
@Test void getProduct_returnsProduct() { // Swap in a fake — no Spring context needed ProductService fakeService = new FakeProductService(); ProductController ctrl = new ProductController(fakeService); // test ctrl.getProduct(1L) ... }

Constructor injection is the preferred style — fields are final, dependencies are explicit, and testing requires no framework magic.

✅
Always use constructor injection — never @Autowired on a field. Constructor injection makes dependencies visible, keeps fields immutable (final), and lets you test without a Spring context. IntelliJ will even warn you if you use field injection.
