STEP 5 — MAVEN & DEPENDENCIES

Your project's
shopping list.

The pom.xml is the one file Maven reads to understand everything about your project — what libraries to download, which Java version to use, how to build the JAR. Learn it once, and dependency management stops being painful.

The file itself

pom.xml — click any section to understand it

Every Spring Boot project has exactly one pom.xml at its root. It has five logical sections. Click each highlighted block to see what it does.

pom.xml
<?xml version="1.0" encoding="UTF-8"?> <project> <modelVersion>4.0.0</modelVersion>
<!-- ① parent --> <parent> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-parent</artifactId> <version>3.3.0</version> </parent>
<!-- ② your project coordinates --> <groupId>com.shop</groupId> <artifactId>my-shop</artifactId> <version>0.0.1-SNAPSHOT</version> <packaging>jar</packaging>
<!-- ③ properties --> <properties> <java.version>21</java.version> </properties>
<!-- ④ dependencies --> <dependencies> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-web</artifactId> </dependency> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-test</artifactId> <scope>test</scope> </dependency> </dependencies>
<!-- ⑤ build plugin --> <build> <plugins> <plugin> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-maven-plugin</artifactId> </plugin> </plugins> </build>
</project>
← Click any highlighted block to understand that section.
Version management

The parent POM — why you never version your starters

The single most important line in your pom.xml is the <parent>. It inherits from spring-boot-starter-parent, which itself inherits from a BOM — a Bill of Materials — that defines the exact version of every library Spring Boot has tested against.

📄
your pom.xml
Declares starters with no version numbers
→
🧩
spring-boot-starter-parent
Inherits from the BOM, sets compiler settings, plugin versions
→
📋
spring-boot-dependencies (BOM)
Declares exact versions for ~300 libraries, all compatibility-tested
Dependency Without parent — you guess With parent — managed for you
spring-boot-starter-web <version>3.3.0</version> no <version> needed ✓
jackson-databind <version>2.17.1</version> no <version> needed ✓
hibernate-core <version>6.5.2.Final</version> no <version> needed ✓
junit-jupiter <version>5.10.3</version> no <version> needed ✓
a non-Spring library (e.g. MapStruct) not in BOM you must specify <version>
⚠️
Only libraries in the Spring Boot BOM are version-managed. For anything outside the BOM — like MapStruct, Lombok, or third-party SDKs — you still need to specify the version yourself. Check start.spring.io/actuator/info to see the full list of what's in the current BOM.
Dependency scopes

When is each dependency available?

Every dependency has a scope — it controls when the library is on the classpath. Getting this wrong either bloats your JAR or causes a runtime error. Click each scope to see when to use it.

compile
Compile + runtime
The default. Available everywhere — compilation, tests, and the final JAR.
test
Tests only
Only on the classpath when running tests. Not included in the final JAR.
runtime
Runtime only
Not needed to compile, but required when the app runs. Classic for JDBC drivers.
provided
Compile only
Needed to compile, but the runtime environment provides it. Not in the JAR.

Adding a dependency

The five-step workflow — every time

You'll follow this exact process every time you need a new library. Click each step to expand it.

01
Decide what you need ›
Be specific. "I need to send emails" → Jakarta Mail. "I need to validate JSON" → it's already included via Hibernate Validator in the web starter. "I need to map between objects" → MapStruct or ModelMapper. Knowing the problem precisely makes the search fast.
02
Find it — two reliable sources ›
For Spring-specific features: go to start.spring.io, search the dependency, and copy the Maven coordinates from the metadata panel. It shows the right starter name and version is handled by the parent.

For everything else: search mvnrepository.com, find your library, pick the latest stable version, and copy the <dependency> snippet.
03
Understand the three coordinates — GAV ›
Every Maven dependency is identified by three coordinates called GAV.
<dependency> <groupId>org.springframework.boot</groupId> <!-- G: who made it (like a package name) --> <artifactId>spring-boot-starter-web</artifactId> <!-- A: the specific library --> <version>3.3.0</version> <!-- V: omit if managed by parent --> </dependency>
Think of it like a file path: groupId/artifactId/version. Maven uses this to locate the exact JAR in Maven Central.
04
Add it to the <dependencies> block ›
Paste the snippet inside the <dependencies> block. Set the correct scope if needed (default is compile). Save the file.
<!-- Example: adding Spring Data JPA + H2 for Step 8 --> <dependency> <groupId>org.springframework.boot</groupId> <artifactId>spring-boot-starter-data-jpa</artifactId> </dependency> <dependency> <groupId>com.h2database</groupId> <artifactId>h2</artifactId> <scope>runtime</scope> <!-- runtime: driver not needed at compile time --> </dependency>
05
Maven downloads it — use it in your code ›
IntelliJ detects the pom.xml change and offers to reload the project. Click "Load Maven Changes" (or hit the Maven refresh icon). Maven downloads the JAR to ~/.m2/repository/ — a local cache so subsequent builds are instant. The library is now on your classpath and available in your code.
// After reloading — auto-configuration kicks in automatically // Spring Boot detects spring-data-jpa on the classpath // and auto-configures EntityManager, DataSource, etc. // Your repositories now work without any manual wiring. @Repository public interface ProductRepository extends JpaRepository<Product, Long> { // findAll(), findById(), save(), delete() — available immediately }
💡
IntelliJ has a built-in Maven tool window (right sidebar). It shows your dependency tree, lets you re-download sources, and has a Dependencies view to see transitive dependencies. If a class is missing at runtime, look there first.
✅
You now own the pom.xml. Five sections, one parent that manages versions, scopes for controlling availability, and a five-step workflow for adding anything new. The next step is project structure — where to put every file, and the four-file pattern behind every feature.
